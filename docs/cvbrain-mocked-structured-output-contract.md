# CVBrain Mocked Structured Output Contract

Date: 2026-06-08

Status: test and documentation contract only. This phase does not implement real OpenAI calls, add the OpenAI dependency, call external services, add a database, change Cloud Run deploy behavior, change `/api/job-intake/analyze`, remove deterministic extraction, add candidate ranking, or introduce candidate data.

## Purpose

A mocked AI structured output is a checked-in JSON payload that represents what a future OpenAI Structured Output response should look like after interpreting recruiter intake text. It is not generated by OpenAI and it is not used by the runtime endpoint.

This phase exists before real OpenAI integration so CVBrain can prove three things safely:

1. Future AI output can express CVBrain Job Intelligence v1.
2. That rich output can be converted back into the current flat WordPress compatibility contract.
3. Guardrails catch country leakage, invented requirements, hidden hard filters, missing clarification policy, and candidate-data exposure before any provider is connected.

The mocked outputs are fixtures, not examples for production storage. They should stay sanitized and contain no recruiter PII, candidate PII, secrets, raw CVs, candidate IDs, names, headlines, emails, phones, addresses, or private logs.

## Why Mocked Output Comes Before OpenAI

The current CVBrain API is deterministic and returns a flat compatibility contract for WordPress Job Intake. A future AI extractor needs to produce a richer object, but the system must not risk changing staging or production behavior while that contract is still being shaped.

Mocked Structured Output lets us validate:

- required Job Intelligence v1 sections;
- recruiter/company clarification behavior;
- search readiness and continue-anyway policy;
- location guardrails for Uruguay, Argentina, and cross-country conflicts;
- hard-filter metadata boundaries;
- flat compatibility mapping back to the existing response shape;
- privacy and no-candidate-data boundaries.

Only after this contract remains stable should a real provider implementation be added behind explicit environment controls.

## Runtime Boundary

The mocked contract layer is library/test-only:

- `app/main.py` is not changed.
- `/health` is not changed.
- `/api/job-intake/analyze` is not changed.
- deterministic extraction remains the runtime behavior.
- no network calls are made.
- no OpenAI package is required.
- no raw mocked payload is returned by the endpoint.

## Guardrails

### Hallucinated Location

Mocked output must preserve explicit source locations and country context separately. It must not:

- infer Argentina, Buenos Aires, CABA, GBA, or AMBA from Uruguay text;
- infer Montevideo, Canelones, or Uruguay from Argentina text;
- convert CABA/GBA into Montevideo;
- convert Montevideo into CABA/GBA;
- treat location as a hard filter unless the source and policy support that later approval.

If source text conflicts with country context, the output should preserve the source location and add a `country_context_mismatch` warning.

### Invented Requirements

Mocked output must not invent:

- industry;
- tools;
- compensation;
- location;
- seniority;
- degree;
- license;
- certification;
- team size;
- travel;
- B2B/B2C context.

Missing but useful details belong in `missing_information` and `company_clarification_questions`, not in `must_have`.

### Hidden Hard Filters

The schema separates:

- `hard_filter_candidate`: this requirement might become a hard filter later if reviewed.
- `hard_filter_approved`: this requirement is approved as a hard filter.

Preferred or nice-to-have items must not become `hard_filter_approved`. Soft/interview-verifiable competencies such as negotiation should become `candidate_screening_questions`, not hidden resume filters.

## Fixture Suite

Mocked outputs live under:

```text
tests/fixtures/mocked_ai_outputs/
```

Required coverage:

- Uruguay Account Manager / medical devices / Montevideo / hybrid;
- Argentina Account Manager / medical devices / Buenos Aires / CABA / GBA;
- Uruguay context with source CABA/GBA mismatch;
- ambiguous clerk search that can continue anyway;
- vague Sales Manager with negotiation as interview-verifiable.

Tests validate each mocked output against the Job Intelligence v1 contract and derive flat compatibility fields:

- `role_title`
- `must_have`
- `should_have`
- `nice_to_have`
- `credentials`
- `experience.minimum_years`
- `experience.seniority`
- `location.normalized`
- `location.hybrid_allowed`
- `search_terms`
- `semantic_terms`
- `recruiter_questions`
- `warnings`
- `confidence`

## Clarification and Search Readiness

`company_clarification_questions` are questions for the recruiter or hiring company. They improve search precision, but they do not block an ordinary ambiguous search by default.

`candidate_screening_questions` are interview or screening prompts. They are not resume hard filters unless a future reviewed policy explicitly approves them.

`search_readiness.status` can be:

- `ready`
- `usable_with_warnings`
- `exploratory`
- `insufficient_for_precise_search`
- `blocked_for_safety_or_technical_reason`

CVBrain should allow proceed for:

- `ready`
- `usable_with_warnings`
- `exploratory`
- `insufficient_for_precise_search`

CVBrain should block only for:

- empty source text;
- unsafe or prohibited request;
- discriminatory or protected filtering request;
- API/security/permission failure;
- unrecoverable technical extraction failure.

Do not block merely because industry, location, compensation, tools, seniority, or clarifying answers are missing.

## Flat Compatibility

The mocked rich object must be convertible into the current flat response shape so WordPress compatibility is preserved. The mapper is intentionally lossy and conservative:

- `job_profile.normalized_role_title` becomes `role_title`;
- requirement groups become flat string arrays;
- location intelligence becomes the flat `location` object;
- company and candidate questions become `recruiter_questions`;
- readiness warnings become flat `warnings`;
- confidence comes from `quality_control.confidence`.

The flat mapper is not candidate search, ranking, scoring, or Super CV logic.

## Future OpenAI Replacement

When real OpenAI Structured Outputs are added later, the provider response should target this same Job Intelligence v1 shape. The flow should be:

```text
source_text
  -> OpenAI Structured Output
  -> Job Intelligence v1 validation
  -> flat compatibility mapping
  -> gated API response
```

Provider integration must remain behind explicit environment configuration and pass the mocked fixture suite before staging use.
